Pricing Sync

Keeps the pricing table in TimescaleDB current. Runs automatically in the background after configure(), and can be triggered manually when a provider changes prices.

How it works

Kostrack ships a CURRENT_PRICING table in kostrack/sync/pricing_sync.py — the authoritative source of model pricing. On startup, configure(sync_pricing=True) (the default) starts a background thread that compares this table against what's in TimescaleDB and inserts any missing or changed entries.

Pricing entries are versioned — old entries get an effective_to date when prices change, and a new row is inserted. This preserves historical cost accuracy even after provider price changes.

Automatic sync (default)

kostrack.configure(
    dsn="postgresql://...",
    sync_pricing=True,   # default — syncs on startup + every 24h
)

Manual sync

from kostrack.sync.pricing_sync import PricingSync

sync = PricingSync(dsn="postgresql://...")
result = sync.run()
print(result)
# {"inserted": 2, "already_current": 14}

Adding a new model

When a provider releases a new model or changes pricing, add it to CURRENT_PRICING in pricing_sync.py and run:

sync = PricingSync(dsn="postgresql://...")
sync.upsert(
    provider="anthropic",
    model="claude-new-model",
    pricing_model="per_token",
    input_rate=3.00 / 1_000_000,
    output_rate=15.00 / 1_000_000,
    cache_read_rate=0.30 / 1_000_000,
    notes="claude-new-model, added March 2026",
)
Historical accuracy

Never delete or update pricing rows directly. Always use upsert() — it closes the old entry with effective_to=now and inserts a new one. This means cost queries against historical data remain accurate even after price changes.
